# Pre-Release Documentation Review Checklist

## Content Review
- [ ] Review every `.md` document in `docs/` and root that discusses the project
- [ ] Look for inconsistencies in logic, naming, or instructions
- [ ] Look for duplication
- [ ] Feature coverage audit — identify features/flows with no documentation
- [ ] Changelog/release notes — verify one exists and reflects what's actually shipping

## Links & References
- [ ] Look for bad links (internal and external)
- [ ] Look for references that should have a link but don't
- [ ] Table of contents accuracy — verify anchors/headings haven't moved or been renamed

## Accuracy
- [ ] Code snippet validation — verify example commands, env vars, and config snippets work as written
- [ ] Version/number consistency — check version numbers, package names, and release tags are consistent across all docs
- [ ] Screenshot/UI reference accuracy — verify any described UI matches the current app state

## Structure & Navigation
- [ ] Document ordering/flow — confirm logical reading order for a new user (install → configure → use)
- [ ] Entry point clarity — root README gives an accurate first impression and orients users to the rest of the docs

## Terminology & Voice
- [ ] Term consistency — same concept always called the same thing throughout
- [ ] Tone/audience consistency — writing assumes a consistent technical level throughout

## Housekeeping
- [ ] Archived/deprecated content — outdated content is clearly marked, removed, or relocated
- [ ] License and attribution — present and accurate
